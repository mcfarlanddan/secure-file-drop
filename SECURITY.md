# Security Model

This document describes the security architecture of Secure File Drop.

## Architecture Security

### Network Layer

```
┌─────────────┐      ┌─────────────┐
│   Browser   │─────▶│ CloudFront  │
└─────────────┘      └─────────────┘
                            │
              ┌─────────────┴─────────────┐
              ▼                           ▼
       ┌─────────────┐             ┌─────────────┐
       │   S3 /www   │             │Lambda (API) │
       └─────────────┘             └─────────────┘
              ▲                           │
              │                           │ presigned URLs
              │                           ▼
       ┌─────────────┐             ┌─────────────┐
       │   Browser   │◀───────────▶│ S3 /uploads │
       └─────────────┘  PUT files  └─────────────┘
```

### Security Controls by Layer

| Layer | Control | Description |
|-------|---------|-------------|
| CDN | CloudFront | HTTPS only, TLS 1.2+ enforced |
| CDN | AWS Shield Standard | DDoS protection (included free with CloudFront) |
| Storage | S3 Block Public Access | All public access blocked |
| Storage | S3 SSE | Server-side encryption (AES-256) |
| Storage | S3 SSL | Bucket policy enforces HTTPS |
| Access | CloudFront OAC | S3 only accessible via CloudFront |
| Compute | Lambda IAM | Least-privilege permissions |
| API | Input Validation | Zod schemas for all endpoints |

## Input Validation

All API endpoints validate input using Zod schemas before processing:

| Field | Validation |
|-------|------------|
| `email` | Valid email format, max 255 chars |
| `fileName` | Sanitized (no path traversal), max 255 chars |
| `fileSize` | Positive integer, max 5TB |
| `title` | Optional, max 500 chars |
| `description` | Optional, max 2000 chars |
| `partNumbers` | Array of positive integers, max 10,000 per request |
| `submissionId` | Valid UUID format |

### Filename Sanitization

The `sanitizeFileName()` function prevents:

- Path traversal attacks (`../../../etc/passwd` → `passwd`)
- Windows path injection (`C:\Users\...` → filename only)
- Hidden files (`.hidden` → `untitled_file`)
- Command injection (`file;rm -rf.sh` → `file_rm -rf.sh`)
- Dangerous characters (`<script>` → `_script_`)

```typescript
// Characters allowed: a-zA-Z0-9._- and space
const sanitized = baseName.replace(/[^a-zA-Z0-9._\- ]/g, '_');
```

## S3 Presigned URLs

Presigned URLs are used for browser-direct uploads to S3:

### How They Work

1. Lambda generates a cryptographically signed URL
2. URL contains: bucket, key, expiry time, signature
3. Browser uploads directly to S3 using the URL
4. S3 validates the signature before accepting the upload

### Security Properties

| Property | Value |
|----------|-------|
| Algorithm | AWS Signature Version 4 |
| Expiry | 1 hour (3600 seconds) |
| Scope | Single PUT operation to specific key |
| Permissions | Only what the Lambda role has |

### Why CORS `*` is Safe

S3 CORS is configured with `allowedOrigins: ['*']`. This is secure because:

1. **Presigned URLs are self-authenticating** - The signature proves authorization
2. **No ambient credentials** - Unlike cookies, presigned URLs don't rely on browser state
3. **Time-limited** - URLs expire after 1 hour
4. **Scoped** - Each URL only works for one specific S3 key

CORS is not a security boundary for presigned URL uploads.

## Content Security Policy

The frontend implements a strict CSP:

```
default-src 'self';
script-src 'self';
style-src 'self' 'unsafe-inline';
connect-src 'self' https://*.amazonaws.com https://*.s3.amazonaws.com;
img-src 'self' data:;
form-action 'self';
frame-ancestors 'none';
base-uri 'self';
upgrade-insecure-requests;
```

### Protections

| Directive | Protection |
|-----------|------------|
| `script-src 'self'` | Prevents XSS via external scripts |
| `frame-ancestors 'none'` | Prevents clickjacking |
| `form-action 'self'` | Prevents form hijacking |
| `base-uri 'self'` | Prevents base tag injection |
| `upgrade-insecure-requests` | Forces HTTPS |

## Request Tracing

All requests include a unique `X-Request-Id` header for:

- Correlating logs across services
- Debugging failed uploads
- Audit trail

Request IDs are:
- Generated by Lambda Function URL (or fallback UUID)
- Included in all log entries (structured JSON)
- Returned in API responses

## CloudWatch Alarms

Operational security monitoring:

| Alarm | Threshold | Description |
|-------|-----------|-------------|
| Lambda Errors | 5 errors / 5min | API failures |
| Lambda Throttles | 1 throttle / 5min | Capacity issues |
| Lambda Duration | p99 > 25s | Approaching timeout |
| CloudFront 5xx | 5% error rate | Origin failures |

All alarms notify via SNS (same topic as upload notifications).

## Threat Model

### Threats Mitigated

| Threat | Mitigation |
|--------|------------|
| Path Traversal | Filename sanitization |
| XSS | CSP, no user-generated HTML |
| Clickjacking | `frame-ancestors 'none'` |
| CSRF | No cookies, no ambient auth |
| DDoS | AWS Shield Standard (included free) |
| Data in Transit | TLS 1.2+ enforced |
| Data at Rest | S3 SSE-S3 encryption |
| Unauthorized Access | S3 Block Public Access, OAC |
| Dangerous File Types | Content-type sanitization (blocks HTML, JS, SVG, etc.) |

### Download Link Security

Download links sent via email are presigned S3 URLs valid for 7 days. Security considerations:

- **Link Exposure**: Anyone with the download link can access the file until it expires
- **No Additional Auth**: Links are self-authenticating (no login required)
- **Email Security**: The link's security depends on the recipient's email security
- **Mitigation**: Links expire after 7 days; files can be manually deleted if needed

For higher security requirements, consider:
- Shorter link expiry (modify `RETENTION_DAYS` constant)
- Adding download authentication (requires architectural changes)
- Enabling S3 Object Lock for compliance scenarios

### Out of Scope

| Concern | Reason |
|---------|--------|
| Authentication | Public upload system by design |
| Virus Scanning | Add S3 Object Lambda or third-party scanner if needed |
| File Content Validation | MIME type is sanitized but file content is not scanned |
| Rate Limiting per IP | Add WAF if needed (extra cost: ~$5/ACL + $1/rule/month) |

**Note**: Content-type validation IS implemented (dangerous MIME types like HTML, JavaScript, and SVG are blocked and replaced with `application/octet-stream`). However, this validates the declared type, not the actual file content. Determined attackers could rename malicious files.

## Production Hardening Checklist

### Already Configured (via CDK)
- [x] Lambda reserved concurrency (optional: `-c reservedConcurrency=10`, requires quota >= 110)
- [x] S3 Block Public Access enabled
- [x] S3 server-side encryption (AES-256)
- [x] S3 lifecycle rules (abort incomplete uploads, transition to Intelligent-Tiering)
- [x] CloudFront Origin Access Control (OAC)
- [x] CloudWatch alarms (Lambda errors, throttles, duration, CloudFront 5xx)
- [x] Structured JSON logging for CloudWatch Insights
- [x] Content-type sanitization (blocks dangerous MIME types)
- [x] Filename sanitization (prevents path traversal)
- [x] Input validation (Zod schemas on all endpoints)

### Included in CloudFront (No Extra Config Needed)
- [x] AWS Shield Standard (DDoS protection)
- [x] TLS certificates

### Manual Configuration Required
- [ ] Confirm SNS subscription in email
- [ ] Test CloudWatch alarms fire correctly
- [ ] Enable S3 access logging (configure target bucket)
- [ ] Enable CloudFront access logging (configure target bucket)
- [ ] Enable CloudTrail for API auditing
- [ ] Configure AWS Config rules for drift detection

### Optional (For Higher Security)
- [ ] Add AWS WAF for rate limiting and bot protection (~$5/ACL + $1/rule/month)
- [ ] Enable S3 Object Lock for compliance (prevents deletion)
- [ ] Configure VPC endpoints for S3/SNS (no internet egress from Lambda)
- [ ] Add virus scanning via S3 Object Lambda
- [ ] Reduce `RETENTION_DAYS` for shorter download link expiry
- [ ] Enable CloudWatch Logs retention policy (cost control)

| filter requestId = 'YOUR-REQUEST-ID'

## Reporting Security Issues

If you discover a security vulnerability:

1. **Do not** open a public GitHub issue
2. Email the maintainer with details
3. Include steps to reproduce
4. Allow reasonable time for a fix
